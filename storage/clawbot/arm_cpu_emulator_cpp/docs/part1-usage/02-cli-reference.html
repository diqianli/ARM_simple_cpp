Usage Syntax

arm_cpu_sim [options] <trace_file>

The trace file is provided as a positional argument. All other parameters are passed as flags. Options can appear in any order relative to the trace file.

Input Control

These options control how the simulator reads and processes input instructions.

-f, --format <type>
Select the input format. Supported values: elf, text, champsim, champsim_xz, json. Default is text. See Input Formats for details on each format.
<trace_file>
Positional argument specifying the path to the input file (ELF binary, text trace, ChampSim trace, or JSON trace). This is the only required argument for running a simulation.
-n, --max-instructions <N>
Maximum number of instructions to simulate. A value of 0 (default) means simulate all instructions in the input. Useful for quick benchmarking or limiting simulation time.
-s, --skip-instructions <N>
Skip the first N instructions from the input before starting the simulation. Useful for skipping initialization code or warm-up phases. Default is 0.
-c, --max-cycles <N>
Maximum number of simulation cycles before the simulator terminates. Acts as a safety limit to prevent infinite loops. Default is 1000000 (1M cycles).

CPU Configuration

These options configure the microarchitectural parameters of the simulated CPU pipeline.

--window-size <N>
Size of the reorder buffer (ROB) / instruction window. This determines how many instructions can be in-flight simultaneously. Larger values allow more instruction-level parallelism but increase resource usage. Default is 128.
--fetch-width <N>
Number of instructions fetched per cycle from the instruction stream. Default is 8. Typical values range from 4 (in-order cores) to 8 (wide superscalar).
--issue-width <N>
Maximum number of instructions issued to execution units per cycle. Default is 4. This is the peak issue bandwidth of the scheduler.
--commit-width <N>
Maximum number of instructions committed per cycle from the ROB. Default is 4. Usually matches the issue width in practical configurations.
Presets override individual settings

Using --preset <name> will override any individually specified CPU and memory configuration parameters. Apply presets first, then override specific values if needed.

Memory Configuration

These options configure the cache hierarchy and memory subsystem timing.

--l1-size <KB>
L1 data cache size in kilobytes. Default is 64 KB. The L1 cache is the closest cache to the CPU core with the lowest latency.
--l2-size <KB>
L2 cache size in kilobytes. Default is 512 KB. The L2 cache serves as a mid-level cache between L1 and L3.
--l3-size <KB>
L3 (last-level) cache size in kilobytes. Default is 8192 KB (8 MB). The L3 cache is shared across cores in multi-core configurations.
--frequency <MHz>
CPU clock frequency in MHz. Default is 2000 MHz (2 GHz). This value is used for calculating timing metrics and power estimates.

Output Control

These options control what output the simulator produces and where it is written.

-o, --output <path>
Path for the output JSON file. This file contains simulation statistics, pipeline trace data, and Konata visualization data. Default is output.json in the current working directory.
-j, --json-metrics
Also emit a summary of performance metrics as JSON to standard output (stdout). Useful for programmatic consumption and scripting pipelines.
--save-trace
Enable saving of the detailed execution trace in the output file. This includes per-instruction timing information and pipeline stage transitions.
--dump-range <start>:<end>
Dump the pipeline trace only for instructions in the specified range (inclusive). Format is start:end. For example, --dump-range 100:200 dumps instructions 100 through 200. Useful for debugging specific instruction sequences.

Visualization

These options control the visualization and web server features.

-p, --port <port>
Start a built-in HTTP server on the specified port to serve the Konata visualization interface. Open http://localhost:<port> in your browser to view the pipeline timeline. The visualization page loads automatically with the simulation output.
-e, --event-mode
Enable event-driven (SimulationEngine) mode. In this mode, the simulator emits SimulationEvent objects to pluggable sinks instead of running the full CPUEmulator path. This disables Konata visualization export and is intended for analytical use cases.
--viz-format <fmt>
Select the visualization output format. Supported values: konata (Konata pipeline timeline JSON), perf (performance statistics format). Default format is Konata.

Multi-Instance

These options enable running multiple independent simulation instances in parallel, useful for design space exploration.

--multi-instance
Enable multi-instance simulation mode. Each instance runs with its own independent configuration and input. Requires specifying --num-instances and providing separate input files.
--num-instances <N>
Number of simulation instances to run. Default is 4. Only effective when --multi-instance is enabled.
--num-threads <N>
Number of threads to use for parallel execution. Default is auto-detected based on available CPU cores. Can be set explicitly for finer control.

Presets

--preset <name>
Apply a named configuration preset that sets CPU pipeline, cache, and memory parameters in one step. Available presets:
Preset Name Description
default Balanced configuration: 128-entry ROB, 8-wide fetch, 4-wide issue/commit, 64 KB L1, 512 KB L2, 8 MB L3
high_performance Aggressive out-of-order configuration with larger windows and caches for maximum IPC
minimal Minimal configuration with small structures, suitable for fast simulation turnaround
gem5_o3_arm_v7a Configuration matching gem5's O3 ARM v7a CPU model for cross-simulator comparison
Combining presets with individual options

Preset values are applied first, then any explicitly specified flags override the corresponding preset values. For example, --preset high_performance --l1-size 32 uses the high_performance preset but overrides L1 size to 32 KB.

Other Options

-m, --memory-size <MB>
Total simulated memory size in megabytes for ELF simulation. Default is 256 MB. This is the size of the simulated physical address space available to the loaded ELF program. Increase this value if your program accesses large memory regions.
-t, --threads <N>
DEPRECATED Use --num-threads <N> instead. This flag is retained for backward compatibility but may be removed in a future version.
-v, --verbose
Enable verbose output. Prints additional information during simulation including per-cycle status, cache events, and pipeline stage transitions. Useful for debugging and understanding simulator behavior.

Example Commands

Basic ELF simulation

./build/arm_cpu_sim -f elf tests/data/test_elf_aarch64

Limited simulation with custom output

./build/arm_cpu_sim -f elf program.elf -n 10000 -o results/run1.json

High-performance CPU configuration

./build/arm_cpu_sim -f elf program.elf --preset high_performance -o results/hp.json

Custom pipeline with visualization server

./build/arm_cpu_sim -f elf program.elf \
  --window-size 256 --issue-width 6 --commit-width 6 \
  --l1-size 128 --l2-size 1024 \
  -p 8080 -o results/custom.json

Event-driven mode with JSON metrics

./build/arm_cpu_sim -e -f elf program.elf -j

Debug specific instruction range

./build/arm_cpu_sim -f elf program.elf --dump-range 50:100 -v
