Running Simulations

Learn how to execute simulations with various options, control simulation scope, and run parallel multi-instance experiments.

Basic Simulation

The simplest way to run the simulator is to pass an ELF binary:

# Input file: tests/data/test_elf_aarch64
# Output file: output.json (default, in current directory)
./build/arm_cpu_sim -f elf tests/data/test_elf_aarch64

Use -o to specify a custom output path:

# Input file: program.elf
# Output file: result.json
./build/arm_cpu_sim -f elf program.elf -o result.json

Visualization Server

Start a built-in HTTP server that serves the Konata pipeline visualization page:

./build/arm_cpu_sim -f elf program.elf -p 3000

This starts an HTTP server on port 3000. Open http://localhost:3000 in your browser to view the interactive pipeline timeline. The output JSON data is served alongside the visualization page.

Event-Driven Mode

Use the -e flag to run in event-driven mode via SimulationEngine. This mode skips Konata visualization export and is ideal for batch processing or when you only need metrics:

./build/arm_cpu_sim -e -f elf program.elf

Limiting Simulation Scope

Control how many instructions or cycles are simulated with these flags:

Flag Description Example
-n Max instructions to simulate -n 1000 — simulate only the first 1000 instructions
-s Skip first N instructions -s 500 — skip the first 500 instructions (warm-up)
-c Max cycles to simulate -c 500000 — stop after 500000 cycles
Warm-up with -s

Use -s to skip a warm-up phase (e.g., -s 1000 -n 5000) so that metrics reflect steady-state behavior rather than cold-start effects.

Multi-Instance Parallel Simulation

Run multiple independent simulation instances in parallel, each with its own configuration. This is ideal for design space exploration:

./build/arm_cpu_sim -e --multi-instance --num-instances 4 --num-threads 4 -f elf program.elf
  • Each instance receives an independent CPUConfig
  • Results are written to separate output files
  • Use --num-threads to control parallelism (defaults to hardware concurrency)
  • Combine with --preset or individual --* flags to vary parameters per instance

JSON Metrics Output

The -j flag emits a compact metrics summary as JSON to stdout, separate from the full output file:

./build/arm_cpu_sim -f elf program.elf -j

This is useful for scripting and automated analysis pipelines where you need quick access to IPC, cache hit rates, and other aggregate metrics.

Combining Options

Multiple flags can be combined freely. For example:

# Input file: program.elf
# Output file: result.json
# Limit to 10000 instructions and 100000 cycles, with JSON metrics on stdout
./build/arm_cpu_sim -f elf program.elf -n 10000 -c 100000 -o result.json -j

Typical Workflow Examples

1. Quick Test

Run a fast simulation with limited scope to verify correctness:

./build/arm_cpu_sim -f elf tests/data/test_elf_aarch64 -n 100 -c 10000

2. Full Simulation with Visualization

Run the complete simulation and launch the web-based visualization server:

./build/arm_cpu_sim -f elf program.elf -p 3000

3. Batch Processing

Run in event-driven mode with JSON metrics output for automated pipelines:

./build/arm_cpu_sim -e -f elf program.elf -j -o output.json

4. Design Space Exploration

Run multiple parallel instances with independent configurations:

./build/arm_cpu_sim -e --multi-instance --num-instances 4 -f elf program.elf
Always report file paths

When reporting simulation results, always specify both the input file path and the output file path so that results are fully reproducible.
