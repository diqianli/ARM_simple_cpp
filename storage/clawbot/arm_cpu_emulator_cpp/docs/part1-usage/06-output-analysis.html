Output & Analysis

Understand the JSON output format, performance metrics, and how to interpret simulation results.

JSON Output File

The simulator writes results to a JSON file (default: output.json, configurable with -o). The output contains three main sections:

1. Metadata

A snapshot of the CPU configuration used for the simulation, including:

  • window_size (ROB size)
  • fetch_width, issue_width, commit_width
  • Cache sizes and associativities (L1, L2, L3)
  • Cache latencies
  • CPU frequency
  • Load/store queue sizes and pipeline counts

2. Operations

Per-instruction pipeline timing data. Each entry records the cycle at which the instruction passed through each pipeline stage:

Field Description
fetch_cycleCycle when instruction was fetched
decode_cycleCycle when instruction was decoded
rename_cycleCycle when register renaming completed
dispatch_cycleCycle when instruction was dispatched to issue queue
issue_cycleCycle when instruction was issued to a functional unit
execute_cycleCycle when execution began
memory_cycleCycle when memory access completed (load/store only)
writeback_cycleCycle when result was written back
commit_cycleCycle when instruction retired (committed)

3. Lanes

Pipeline stage details per cycle, used by the Konata visualization to render the timeline chart.

Performance Metrics

Performance metrics are emitted with the -j flag (to stdout) or included in the output JSON. The key metrics are:

IPC (Instructions Per Cycle)

total_instructions / total_cycles. Higher is better. A well-optimized out-of-order core typically achieves IPC > 1.0 for non-memory-bound workloads.

CPI (Cycles Per Instruction)

Inverse of IPC. Lower is better. Useful for comparing against in-order processors.

Cache Hit Rate

hits / (hits + misses) for each cache level (L1, L2, L3). A high L1 hit rate (> 90%) is critical for performance.

MPKI (Misses Per Kilo Instructions)

cache_misses * 1000 / total_instructions. Normalized metric for comparing cache performance across different simulation lengths.

Branch Prediction Accuracy

correct_predictions / total_branches. Modern branch predictors achieve > 95% accuracy on typical workloads.

Functional Unit Utilization

Breakdown of how busy each functional unit type was during the simulation:

  • IntAlu — Integer arithmetic and logic
  • IntMul — Integer multiply and divide
  • Load — Load units
  • Store — Store units
  • Branch — Branch resolution
  • FpSimd — Floating-point and SIMD
  • System — System instructions (barriers, etc.)
  • Crypto — Cryptographic instructions

Pipeline Stall Distribution

Breakdown of pipeline stalls by type:

  • Data hazard — Waiting for a register dependency
  • Control hazard — Waiting for branch resolution
  • Structural hazard — No available functional unit
  • Memory — Waiting for cache or memory access

Time Series

Per-1000-cycle sampling of IPC and other metrics, useful for identifying performance phase changes over the course of a simulation.

Detailed Trace Options

Save Execution Trace

Use --save-trace to save a detailed execution trace including per-cycle state of all pipeline stages:

./build/arm_cpu_sim -f elf program.elf --save-trace

Dump Specific Instruction Range

Use --dump-range to dump the pipeline trace for a specific instruction range:

# Dump pipeline trace for instructions 100 through 200
./build/arm_cpu_sim -f elf program.elf --dump-range 100:200

Tips for Analyzing Results

Low IPC + high L1 miss rate

The workload is memory-bound. Consider increasing the L1 cache size, improving prefetching, or optimizing the data access pattern in the test program.

Low IPC + low branch accuracy

The workload is branch-heavy with hard-to-predict branches. Consider branch predictor improvements or reducing branch density in the test program.

High FU utilization imbalance

The workload has specific resource bottlenecks. For example, if the load units are saturated but IntAlu is idle, the program is limited by memory bandwidth rather than compute throughput.
