Writing Test Programs

How to write bare-metal ARM64 assembly and C programs for the simulator, covering instruction patterns, compilation, and execution environment details.

Bare-Metal ARM64 Programming Basics

The simulator executes ARM64 ELF binaries in a bare-metal environment. There is no operating system, no libc, and no system call interface. Your program runs directly on the simulated hardware.

Entry Point

The entry point must be the _start symbol. The ELF loader begins execution at whatever address this symbol resolves to. If _start is not found, the loader uses the ELF entry point from the file header.

.global _start
_start:
    // Your code begins here
    MOV X0, #42
    // ...

ELF Sections

Section Contents Description
.text Executable code Instructions loaded into simulated memory and executed by the pipeline
.data Initialized data Variables with explicit initial values, loaded from the ELF file
.bss Zero-initialized data Variables that start at zero, allocated but not stored in the ELF file

Register Conventions (AAPCS64)

Registers Role Caller/Callee Saved
X0 – X7 Arguments and return values Caller-saved
X8 Indirect result location register Caller-saved
X9 – X15 Temporary / scratch registers Caller-saved
X16 – X17 Intra-procedure call scratch registers Caller-saved
X18 Platform register (reserved) Varies by platform
X19 – X28 Callee-saved registers Callee-saved
X29 Frame pointer (FP) Callee-saved
X30 Link register (LR) Caller-saved
SP Stack pointer Special
XZR (WZR) Zero register (always reads as 0) N/A
Program termination

There is no exit() syscall available. The simulator terminates when the max instruction count (-n) or max cycle count (-c) is reached. Design your test programs to run within these limits.

Common Instruction Patterns

Arithmetic Instructions

ADD X0, X1, X2      // X0 = X1 + X2
ADD X0, X1, #16     // X0 = X1 + 16 (immediate operand)
SUB X3, X4, #16     // X3 = X4 - 16 (immediate operand)
SUB X3, X4, X5      // X3 = X4 - X5
MUL X5, X6, X7      // X5 = X6 * X7
AND X8, X9, X10     // X8 = X9 & X10 (bitwise AND)
ORR X8, X9, X10     // X8 = X9 | X10 (bitwise OR)
EOR X8, X9, X10     // X8 = X9 ^ X10 (bitwise XOR)
LSL X8, X9, #3      // X8 = X9 << 3 (logical shift left)
LSR X8, X9, #5      // X8 = X9 >> 5 (logical shift right)
ASR X8, X9, #2      // X8 = X9 >> 2 (arithmetic shift right)
MOV X0, X1           // X0 = X1 (register move)
MOV X0, #42          // X0 = 42 (immediate move)
MVN X0, X1           // X0 = ~X1 (bitwise NOT)

Memory Access Instructions

// Load instructions
LDR X0, [X1, #0]    // Load 64-bit value from address X1+0 into X0
LDR X0, [X1, #8]    // Load 64-bit value from address X1+8 into X0
LDR W0, [X1, #4]    // Load 32-bit value from address X1+4 into W0
LDR X0, [X1, X2]    // Load 64-bit value from address X1+X2 (register offset)
LDUR X0, [X1, #-1]  // Load with unscaled offset (supports negative offsets)

// Store instructions
STR X2, [X3, #8]    // Store 64-bit X2 to address X3+8
STR W2, [X3, #4]    // Store 32-bit W2 to address X3+4
STUR X0, [X1, #-1]  // Store with unscaled offset

// Load/Store pair (two registers in one instruction)
LDP X0, X1, [X2]    // Load X0 from [X2], X1 from [X2+8]
LDP X0, X1, [X2, #16] // Load X0 from [X2+16], X1 from [X2+24]
STP X0, X1, [X2]    // Store X0 to [X2], X1 to [X2+8]
STP X0, X1, [X2, #16] // Store X0 to [X2+16], X1 to [X2+24]

// Pre-index and post-index
LDR X0, [X1, #8]!   // Load from X1+8, then X1 = X1+8 (pre-index)
LDR X0, [X1], #8    // Load from X1, then X1 = X1+8 (post-index)

Branch Instructions

B label              // Unconditional branch to label
BL func              // Branch with link (saves return address in X30)
RET                  // Return to address in X30

// Compare and branch
CBZ X0, label        // Branch to label if X0 == 0
CBNZ X0, label       // Branch to label if X0 != 0

// Conditional branches (after CMP or SUBS)
B.EQ label           // Branch if equal (Z flag set)
B.NE label           // Branch if not equal (Z flag clear)
B.LT label           // Branch if less than (signed)
B.GT label           // Branch if greater than (signed)
B.LE label           // Branch if less than or equal (signed)
B.GE label           // Branch if greater than or equal (signed)
B.HI label           // Branch if unsigned higher
B.LS label           // Branch if unsigned lower or same

// Compare instruction (sets flags without writing result)
CMP X0, X1           // Sets flags based on X0 - X1
CMP X0, #42          // Sets flags based on X0 - 42

// Conditional select
CSEL X0, X1, X2, EQ  // X0 = (EQ) ? X1 : X2

Floating-Point Instructions

FADD D0, D1, D2      // Double-precision: D0 = D1 + D2
FSUB D0, D1, D2      // Double-precision: D0 = D1 - D2
FMUL D0, D1, D2      // Double-precision: D0 = D1 * D2
FDIV D0, D1, D2      // Double-precision: D0 = D1 / D2
FADD S0, S1, S2      // Single-precision: S0 = S1 + S2
FMUL S0, S1, S2      // Single-precision: S0 = S1 * S2
FMOV D0, X0          // Move integer to FP register (reinterpret bits)
FMOV X0, D0          // Move FP register to integer register

SIMD Instructions

// 128-bit vector operations (NEON)
ADD V0.4S, V1.4S, V2.4S  // 4 x 32-bit integer addition
ADD V0.2D, V1.2D, V2.2D  // 2 x 64-bit integer addition
SUB V0.8B, V1.8B, V2.8B  // 8 x 8-bit integer subtraction
MUL V0.4S, V1.4S, V2.4S  // 4 x 32-bit integer multiplication

// SIMD load/store
LDR Q0, [X1]         // Load 128-bit vector from memory
STR Q0, [X2]         // Store 128-bit vector to memory

Memory Barrier Instructions

DMB ISH              // Data Memory Barrier (Inner Shareable)
                       // Ensures memory accesses are observed in order
                       // by other observers in the Inner Shareable domain

DSB ISH              // Data Synchronization Barrier (Inner Shareable)
                       // Blocks execution until all memory accesses complete

ISB                  // Instruction Synchronization Barrier
                       // Flushes the instruction pipeline, ensuring
                       // subsequent instructions are fetched after the barrier

Complete Example Program

The following example demonstrates a simple loop that computes a sum in memory:

.global _start

.section .data
array:
    .quad 10, 20, 30, 40, 50    // 5 elements
size = (. - array) / 8           // Number of elements

.section .bss
result:
    .skip 8                      // 64-bit result variable

.section .text
_start:
    LDR X0, =array               // X0 = address of array
    LDR X1, =size                // X1 = number of elements
    LDR X2, =result              // X2 = address of result
    MOV X3, #0                   // X3 = sum accumulator = 0
    MOV X4, #0                   // X4 = loop index = 0

loop:
    CMP X4, X1                   // Compare index with size
    B.GE done                    // If index >= size, exit loop

    LDR X5, [X0, X4, LSL #3]    // Load array[index] (X4 * 8 bytes offset)
    ADD X3, X3, X5               // sum += array[index]
    ADD X4, X4, #1               // index++
    B loop                       // Repeat

done:
    STR X3, [X2]                 // Store result to memory
    B done                       // Spin forever (simulator terminates via -n or -c)

Compilation

Using the Provided Script (Recommended)

The project includes a convenience script that auto-detects the cross-compiler and compiles the built-in test source:

./scripts/compile_test_elf.sh

This script searches for aarch64-elf-gcc, aarch64-linux-gnu-gcc, or aarch64-none-elf-gcc in your PATH and compiles scripts/test_elf_source.c into tests/data/test_elf_aarch64.

Manual Compilation

# macOS (using Homebrew's aarch64-elf-gcc)
aarch64-elf-gcc -static -O2 -nostdlib -o program program.s

# Linux (using aarch64-linux-gnu-gcc)
aarch64-linux-gnu-gcc -static -O2 -nostdlib -o program program.s

# macOS/Linux (using aarch64-none-elf-gcc)
aarch64-none-elf-gcc -static -O2 -nostdlib -o program program.s

# Compile C source directly (bare-metal)
aarch64-elf-gcc -static -O2 -nostdlib -o program program.c
Compiler flags explained
  • -static — Static linking (no dynamic library dependencies)
  • -O2 — Optimization level 2 (produces typical real-world instruction patterns)
  • -nostdlib — Do not link standard library (no libc, no startup code)

Optimization Levels for Testing

Flag Effect Use Case
-O0 No optimization, literal instruction mapping Verify individual instruction behavior
-O1 Basic optimizations Light testing with some compiler transformations
-O2 Standard optimizations Recommended for realistic simulation workloads
-O3 Aggressive optimizations including vectorization Maximum ILP and SIMD usage testing

Built-in Test Source

The project includes a comprehensive test source file at scripts/test_elf_source.c that covers all major instruction categories:

  • ALU operations — ADD, SUB, MUL, AND, ORR, EOR, shifts, moves
  • Load/Store — LDR, STR, LDP, STP with various addressing modes
  • Branches — conditional branches, compare-and-branch, function calls
  • Floating Point — FADD, FSUB, FMUL, FDIV in single and double precision
  • SIMD — Vector arithmetic with NEON instructions
  • Barriers — DMB, DSB, ISB memory and instruction synchronization barriers

This file is compiled by scripts/compile_test_elf.sh and the resulting binary is placed at tests/data/test_elf_aarch64.

Important Notes

Simulated memory space

Programs run in a simulated physical address space. The default size is 256 MB, configurable with -m <MB>. All code, data, and stack must fit within this space. If your program accesses memory beyond the allocated range, the simulator will report a memory error.

No OS services

This is pure bare-metal execution. There are no system calls, no printf, no file I/O, no timers, and no interrupt handling. All data must be initialized statically or computed at runtime within the program.

Stack pointer setup

The stack pointer (SP) is initialized to the top of the simulated memory region by the ELF loader. However, if your program uses deeply nested function calls or large stack frames, ensure the stack has sufficient room. You can also manually set SP to a specific address in your _start routine.

Termination strategy

Since there is no exit mechanism, design your test programs with an infinite loop at the end (B .) and use -n <N> or -c <N> to limit simulation length. Alternatively, use a finite loop that naturally exhausts the instruction count.

Running Your Program

# Compile
aarch64-elf-gcc -static -O2 -nostdlib -o my_program my_program.s

# Simulate (input: my_program, output: output.json)
./build/arm_cpu_sim -f elf my_program -o output.json

# Simulate with limited instructions and visualization
./build/arm_cpu_sim -f elf my_program -n 10000 -p 3000 -o output.json
