CHI Coherence Protocol

Implementation of the AMBA CHI (Coherent Hub Interface) cache coherence protocol for multi-core cache consistency modeling.

The simulator implements the AMBA CHI (Coherent Hub Interface) cache coherence protocol, providing a realistic model of cache coherence traffic in ARM-based multi-core systems. The implementation is in chi/chi_manager.hpp.

Disabled by default

CHI coherence is disabled by default for single-core simulations. Enable it with --enable-chi on the command line or set CPUConfig::enable_chi = true programmatically.

CHI Node Types

The CHI protocol models three node types, each representing a different role in the coherence hierarchy:

1. RN-F (Requesting Node - Fully coherent)

Represents the CPU core's cache controller. It issues read and write requests to the interconnect and handles the following transactions:

  • Role: Issues read/write requests to the interconnect
  • Node ID: Configurable via chi_rnf_node_id (default: 0)
  • Request types: ReadShared, ReadClean, ReadNotSharedDirty, WriteLineUnique, MakeUnique

2. HN-F (Home Node - Fully coherent)

Acts as the directory-based coherence manager, maintaining cache line state information:

  • Role: Directory-based coherence manager
  • Maintains: Cache line state directory
  • Handles: Snoops and data responses
  • Node ID: Configurable via chi_hnf_node_id (default: 1)
  • Directory size: Configurable via chi_directory_size (default: 4096 entries)

3. SN-F (Snoop Node - Fully coherent)

Handles snoop requests from the HN-F and provides data for cache-to-cache transfers:

  • Role: Handles snoop requests from HN-F
  • Provides: Data for cache-to-cache transfers
  • Node ID: Configurable via chi_snf_node_id (default: 2)

Protocol State Machine

Cache lines transition through CHI coherence states as requests are processed:

State Description
Invalid Line not present in the cache
Shared (S) Clean copy; may exist in multiple caches simultaneously
Unique (U) Only copy in the system; clean
UniqueDirty (UD) Only copy in the system; modified (dirty)

CHI Configuration Parameters

Parameter Default Description
enable_chi false Enable CHI protocol
chi_request_latency 2 cycles Request transmission latency
chi_response_latency 2 cycles Response transmission latency
chi_data_latency 4 cycles Data transfer latency
chi_snoop_latency 2 cycles Snoop round-trip latency
chi_max_pcrd_credits 16 Max PCR (Protocol Credit) credits
chi_max_outstanding_dbid 32 Max outstanding data buffer IDs
chi_max_retry_queue_size 64 Max retry queue entries
chi_directory_size 4096 Directory entries

CHI Request Types

The protocol supports the following request types, each representing a specific coherence transaction:

enum class ChiRequestType {
    ReadShared,         // Read for sharing
    ReadClean,          // Read expecting clean data
    ReadNotSharedDirty, // Read with intent to modify
    WriteLineUnique,    // Write full cache line
    MakeUnique,         // Upgrade to unique
    CleanShared,        // Evict shared line
    MakeInvalid,        // Evict unique/dirty line
};

QoS (Quality of Service)

The CHI implementation includes flow control mechanisms to manage interconnect bandwidth:

  • PCR credit-based flow control: Limits the number of in-flight requests using Protocol Credit (PCR) credits, preventing interconnect congestion.
  • Outstanding request limiting: Caps the number of concurrent data buffer IDs (chi_max_outstanding_dbid) to bound resource usage.
  • Retry queue for back-pressure: When credits are exhausted or the interconnect is congested, requests are placed in a retry queue (chi_max_retry_queue_size) and re-issued when bandwidth becomes available.
