Type System & Data Structures

Core types, enums, and data structures used throughout the ARM CPU Emulator, including register definitions, instruction representation, and error handling.

All core types are defined in include/arm_cpu/types.hpp (732 lines). The file uses modern C++20 features including scoped enums, structs, constexpr, and concepts.

Register Types

The simulator models the ARM64 register files with strongly-typed enums:

// General purpose registers
enum class Reg : uint8_t { X0, X1, ..., X28, X29, FP, LR, SP, XZR };

// Vector/SIMD registers
enum class VReg : uint8_t { V0, V1, ..., V31 };

// Predicate registers (SVE)
enum class PReg : uint8_t { P0, P1, ..., P15 };
  • Reg: 31 general-purpose registers (X0–X30) plus frame pointer (FP/X29), link register (LR/X30), stack pointer (SP), and zero register (XZR).
  • VReg: 32 128-bit SIMD/NEON registers (V0–V31).
  • PReg: 16 predicate registers for SVE (P0–P15).

OpcodeType Enum (~80 entries)

The central enum classifying every instruction the simulator recognizes. Grouped by functional category:

enum class OpcodeType : uint8_t {
    // Integer ALU
    Add, Sub, And, Or, Xor, Shift, Rotate, Neg, Not,
    Mul, Mneg, Div, Udiv, Rem, Urem,
    // Comparison
    Cmp, Cmn, Tst,
    // Move
    Mov, Mvn, Movz, Movk,
    // Load/Store
    Load, Store, LoadPair, StorePair, LoadByte, StoreByte,
    LoadHalf, StoreHalf, LoadSigned, Prefetch,
    // Branch
    Branch, BranchLink, BranchReg, CondBranch,
    CompareBranch, TestBranch, Return, ExceptionReturn,
    // Floating Point
    FpAdd, FpSub, FpMul, FpDiv, FpSqrt,
    FpCmp, FpConvert, FpRound,
    // SIMD/NEON
    SimdAdd, SimdSub, SimdMul, SimdShift, SimdCompare,
    SimdLoad, SimdStore, SimdMove,
    // SVE
    SveAdd, SveSub, SveMul, SveLoad, SveStore,
    // SME
    SmeAdd, SmeMul, SmeLoad, SmeStore, SmeOuterProduct,
    // Crypto
    AesEncode, AesDecode, AesMix, AesMc,
    Sha1Hash, Sha256Hash, Sha512Hash,
    // System
    Barrier, Nop, Sys, Msr, Mrs,
    // Other
    Unknown, Count
};

Classification Functions

Utility functions classify opcodes into categories, used by the pipeline scheduler and statistics collector:

bool is_memory_op(OpcodeType);   // Load, Store, LoadPair, StorePair, etc.
bool is_branch(OpcodeType);      // Branch, BranchLink, CondBranch, etc.
bool is_compute(OpcodeType);     // Add, Sub, Mul, etc.
bool is_simd(OpcodeType);        // SimdAdd, SimdSub, etc.
bool is_sve(OpcodeType);         // SveAdd, SveSub, etc.
bool is_sme(OpcodeType);         // SmeAdd, SmeMul, etc.
bool is_crypto(OpcodeType);      // AesEncode, Sha1Hash, etc.
bool is_barrier(OpcodeType);     // Barrier

Instruction Struct (Builder Pattern)

The central data structure representing a single instruction in the simulator pipeline. Uses a builder pattern for ergonomic construction:

struct Instruction {
    uint64_t pc = 0;
    OpcodeType opcode = OpcodeType::Unknown;
    std::string mnemonic;

    // Registers
    StaticVector<Reg, 4> src_regs;
    StaticVector<Reg, 4> dst_regs;
    StaticVector<VReg, 4> src_vregs;
    StaticVector<VReg, 4> dst_vregs;

    // Memory
    uint64_t mem_addr = 0;
    std::size_t mem_size = 0;
    bool is_load = false;
    bool is_store = false;

    // Branch
    uint64_t branch_target = 0;
    bool branch_taken = false;

    // Builder methods
    Instruction& with_pc(uint64_t pc);
    Instruction& with_opcode(OpcodeType op);
    Instruction& with_src_reg(Reg r);
    Instruction& with_dst_reg(Reg r);
    Instruction& with_mem_access(uint64_t addr, std::size_t size, bool is_load);
    Instruction& with_branch(uint64_t target, bool taken);
};

InstrStatus Enum

Tracks the lifecycle of an instruction as it progresses through the out-of-order pipeline:

enum class InstrStatus : uint8_t {
    Waiting,     // In window, waiting for dependencies
    Ready,       // Dependencies resolved, ready to issue
    Executing,   // Currently executing on functional unit
    Completed,   // Execution finished, result available
    Committed    // Retired from pipeline
};

StaticVector<T, N>

A stack-allocated small vector that avoids heap allocation. Used throughout for register lists and dependency sets:

  • Fixed capacity N: Maximum number of elements (compile-time constant)
  • Dynamic size up to N: Can hold 0 to N elements
  • No heap allocation: All storage is on the stack, suitable for hot-path data structures
  • Typical usage: StaticVector<Reg, 4> for register operand lists

Error Handling: Result<T>

The simulator uses a Rust-style error handling pattern based on std::variant:

// Rust-style error handling
template<typename T>
using Result = std::variant<T, EmulatorError>;

// Success helper
template<typename T>
Result<T> Ok(T value);

// Error helper
template<typename T>
Result<T> Err(EmulatorError error);

// TRY macro for early return on error
#define TRY(expr) \
    ({ auto _result = (expr); \
       if (std::holds_alternative<EmulatorError>(_result)) \
           return std::get<EmulatorError>(_result); \
       std::get<T>(std::move(_result)); })

This pattern propagates errors up the call stack without exceptions, following the same convention used in the original Rust implementation. The TRY macro unwraps the success value or returns the error immediately, similar to Rust's ? operator.
