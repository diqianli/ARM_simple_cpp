Out-of-Order Engine

The core of the simulator implementing out-of-order execution with speculative execution support.

The OoO engine is the core of the simulator, implementing out-of-order execution with speculative execution support.

Architecture Overview

Three main components work together to enable out-of-order execution:

  1. Instruction Window (ooo/window.hpp): The Reorder Buffer (ROB) that holds all in-flight instructions
  2. Dependency Tracker (ooo/dependency.hpp): Tracks register and memory dependencies
  3. Scheduler (ooo/scheduler.hpp): Selects ready instructions for execution

Instruction Window (ROB)

The instruction window is the central data structure of the OoO engine:

  • Size configurable via window_size (default: 128)
  • Each entry is a WindowEntry with:
    • Instruction data
    • InstrStatus state machine: Waiting → Ready → Executing → Completed → Committed
    • Physical register mappings
    • Dependency counters (remaining register + memory dependencies)
  • Instructions enter at the tail and commit from the head (in-order commit)
  • When the window is full, fetch stalls

Dependency Tracking

Dependencies are tracked at two levels:

Register Dependencies

  • Producer-consumer relationships via register renaming
  • Each instruction tracks which physical registers it produces (destination) and consumes (sources)
  • Dependencies resolved when producer completes writeback

Memory Dependencies

  • Load-after-Store (RAW): Load must wait for preceding store to the same address
  • Store-after-Store (WAW): Stores must complete in program order
  • Address-based disambiguation for load/store ordering

Scheduler

The scheduler examines all instructions in the window each cycle and performs the following steps:

  1. Examines all Waiting instructions in the window each cycle
  2. Wakes up instructions whose dependencies are satisfied (transition to Ready)
  3. Selects up to issue_width Ready instructions per cycle
  4. Priority: oldest ready instructions first (in program order)

Execution Unit Types

Unit Type Description Typical Latency
IntAlu Integer arithmetic/logic 1 cycle
IntMul Integer multiply 3 cycles
Load Memory load 4 cycles (L1 hit)
Store Memory store 1 cycle
Branch Conditional/unconditional branch 1 cycle
FpSimd Floating point / SIMD 3–5 cycles
System System instructions (barriers, etc.) 1–10 cycles
Crypto Cryptographic instructions varies
