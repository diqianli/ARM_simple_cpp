GEM5 Comparison

Cross-validation with the GEM5 simulator, including configuration alignment, comparison scripts, and known architectural differences.

The simulator can be configured to match GEM5's O3_ARM_v7a CPU model, enabling cross-validation of timing results between the two simulators. This is useful for verifying correctness and understanding where simplifications in our model affect accuracy.

Configuration Alignment

Use --preset gem5_o3_arm_v7a to align our simulator's parameters with GEM5 defaults:

Parameter Our Simulator GEM5 O3_ARM_v7a
ROB Size 40 40
Fetch Width 3 3
Issue Width 8 8
Commit Width 8 8
Load Queue 16 16
Store Queue 16 16
ICache — 32 KB, 2-way
DCache 32 KB, 2-way 32 KB, 2-way
L2 Cache 1 MB, 16-way 1 MB, 16-way
L1 Hit Latency 2 cycles ~2 cycles
L2 Hit Latency 20 cycles ~20 cycles
Key structural differences

GEM5 O3_ARM_v7a has separate ICache and DCache; our simulator uses a unified L1. GEM5 has no L3 in this configuration; our simulator keeps the default L3. These differences affect cache contention modeling.

Comparison Script

The project includes a Python script scripts/compare_gem5.py that compares output from both simulators side by side:

python3 scripts/compare_gem5.py \
    --ours output.json \
    --gem5 gem5_stats.txt

Comparison Dimensions

The comparison script evaluates the following metrics:

  • Cycle count: Total simulated cycles to completion
  • Instruction count: Total instructions executed
  • IPC (Instructions Per Cycle): Overall throughput metric
  • Execution time: Wall-clock time taken by each simulator

Known Differences

1. Branch Prediction

Our simulator uses FunctionalSim (perfect prediction via pre-execution), while GEM5 uses a configurable branch predictor (default: tournament). This can cause significant IPC differences, as perfect prediction eliminates all branch misprediction penalties.

2. Cache Model

Our L1 cache is unified (instruction + data), while GEM5 has separate ICache and DCache. This affects cache contention modeling, as instruction fetches and data accesses compete for the same cache space in our model but are isolated in GEM5.

3. L3 Cache

Our simulator always includes an L3 cache in the hierarchy, while GEM5 O3_ARM_v7a does not configure an L3. This can lead to different miss rates and memory access patterns.

4. Timing Granularity

Our simulator uses integer cycle counting, while GEM5 may have sub-cycle timing for some operations. This can cause small discrepancies in cycle counts even when the overall behavior is similar.

5. Memory Model

Our DDR model includes row buffer timing and bank conflicts, while GEM5 uses a different memory timing model (typically SimpleMemory or DRAMCtrl). The detailed timing parameters may differ.

6. Instruction Coverage

Our Capstone-based decoder covers ~500 ARM64 mnemonics, while GEM5's ISA parser is more comprehensive. Rare or privileged instructions that Capstone cannot decode may be handled differently.

Benchmark Programs with GEM5 Variants

The benchmarks/ directory includes _gem5_se variants of each benchmark, pre-configured for GEM5's SE (System-Emulation) mode. These allow direct comparison:

benchmarks/dhrystone/dhrystone_gem5_se
benchmarks/fibonacci/fibonacci_gem5_se
benchmarks/matmul/matmul_gem5_se
# ... and other benchmarks

Tips for Meaningful Comparison

  1. Use the same binary: Our _aarch64 and _gem5_se variants are compiled from the same source, ensuring instruction-level equivalence.
  2. Set the same limits: Configure identical max instructions (-n) and max cycles (-c) limits in both simulators.
  3. Account for branch prediction: Our perfect branch prediction (via FunctionalSim) will typically show higher IPC than GEM5's tournament predictor. Focus on cache behavior for more comparable metrics.
  4. Compare cache metrics: Cache hit rates and MPKI (Misses Per Kilo Instructions) are more comparable between the two simulators than raw IPC, as they are less affected by branch prediction differences.
  5. Use the comparison script: Run scripts/compare_gem5.py for standardized output formatting and metric alignment.
