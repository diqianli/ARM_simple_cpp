CPU Pipeline

The 9-stage out-of-order pipeline that drives instruction execution in the simulator.

The simulator models a 9-stage out-of-order pipeline, processing instructions from fetch through commit with out-of-order execution in the middle stages.

Pipeline Stages

  1. Fetch (IF): Reads instructions from the input source (trace or ELF). Fetches up to fetch_width instructions per cycle.
  2. Decode (DE): Decodes raw instruction bytes into internal Instruction structures. Two decoder paths:
    • Capstone path (ELF input): ~500 mnemonics via Capstone v5
    • Text trace path: ~40 mnemonics via custom parser
  3. Rename (RN): Register renaming to eliminate false dependencies. Maps architectural registers (X0–X30, V0–V31) to physical registers.
  4. Dispatch (DI): Dispatches renamed instructions to the instruction window (ROB). Tracks dependencies between instructions.
  5. Issue (IS): Wake-up and select ready instructions from the window. Uses scoreboarding for dependency tracking. Issues up to issue_width instructions per cycle.
  6. Execute (EX): Instructions execute on functional units. Different units have different latencies:
    • Integer ALU: 1 cycle (ADD, SUB, AND, OR, etc.)
    • Integer Multiply: 3 cycles
    • Load: 4 cycles (L1 hit)
    • Store: 1 cycle (address calculation)
    • Branch: 1 cycle
    • FP/SIMD: 3–5 cycles
    • Crypto: varies
  7. Memory Complete (ME): Load data returns from cache hierarchy. May stall for cache misses.
  8. Writeback (WB): Results are written back and broadcast to dependent instructions in the window.
  9. Commit (RR): Instructions commit in program order. Up to commit_width per cycle. Updates architectural state.

Main Loop Implementation

The pipeline is driven by CPUEmulator::run() in src/cpu.cpp. Each cycle:

  1. Commit completed instructions (in-order)
  2. Write back results
  3. Execute ready instructions
  4. Issue new instructions from window
  5. Dispatch/decode/fetch new instructions
  6. Update statistics

Pipeline Tracking

PipelineTracker records stage entry/exit times for each instruction. The data is exported to Konata JSON format for visualization. Per-instruction timing includes:

Field Description
fetch_cycleCycle when instruction entered the Fetch stage
decode_cycleCycle when instruction entered the Decode stage
rename_cycleCycle when instruction entered the Rename stage
dispatch_cycleCycle when instruction entered the Dispatch stage
issue_cycleCycle when instruction was issued for execution
execute_cycleCycle when instruction began executing
memory_cycleCycle when memory operation completed
writeback_cycleCycle when result was written back
commit_cycleCycle when instruction committed
