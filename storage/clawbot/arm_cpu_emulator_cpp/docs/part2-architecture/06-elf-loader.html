ELF Loader

Parses ELF64 little-endian binaries for ARM64 execution, mapping segments into simulated memory and resolving symbols.

Source files: elf/elf_loader.hpp, elf/elf_loader.cpp

ELF Parsing

The ELF loader performs three stages of parsing:

  • ELF Header — Validates the magic number (\x7fELF), class (64-bit), endianness (little-endian), and machine type (AArch64). Rejects binaries that do not match these criteria.
  • Program Headers — Identifies PT_LOAD segments that need to be mapped into simulated memory. Each program header specifies the virtual address, file offset, and size of the segment.
  • Section Headers — Reads section metadata for symbol resolution, string tables, and debug information.

Segment Loading

PT_LOAD segments are loaded into simulated memory with the following permission handling:

Flag Constant Segment Type
PF_X 0x1 Executable (code segment)
PF_W 0x2 Writable (data segment)
PF_R 0x4 Readable (all segments)

Virtual addresses from the ELF file are mapped directly into the simulator's memory space. The total memory size is configurable via the -m flag (default: 256 MB).

Memory size configuration

Use -m <MB> to set the simulated memory size. If your program accesses large memory regions, increase this value. For example: -m 1024 for 1 GB of simulated memory.

Symbol Resolution

The loader parses symbol tables to enable named lookups and address-to-name mapping:

  • Parses .symtab (static) and .dynsym (dynamic) sections
  • Supports STT_FUNC (function symbols) and STT_OBJECT (data symbols)
  • C++ name demangling for mangled symbols (e.g., _Z3foov → foo())
  • PLT (Procedure Linkage Table) symbol detection for dynamically linked functions
  • Symbol lookup by name and by address

Data Structures

The loader uses the following core data structures:

struct MemorySegment {
    uint64_t vaddr;       // Virtual address
    uint64_t size;        // Segment size
    uint64_t flags;       // Permissions (R/W/X)
    std::vector<uint8_t> data;  // Segment content
};

struct SymbolTable {
    struct Symbol {
        std::string name;
        uint64_t address;
        uint64_t size;
        uint8_t type;      // STT_FUNC, STT_OBJECT, etc.
        uint8_t binding;   // STB_LOCAL, STB_GLOBAL, etc.
    };
    std::vector<Symbol> symbols;
};

Entry Point Detection

The simulator determines the initial program counter (PC) using a priority-based approach:

  1. ELF entry point field — Uses the e_entry field from the ELF header as the primary source
  2. _start symbol fallback — If the entry point is 0, searches for the _start symbol in the symbol table
  3. Dynamic ELF support — For dynamically linked binaries, supports GOT/PLT resolution to find the actual entry point

Functional Simulation (FunctionalSim)

Source files: input/functional_sim.hpp, input/functional_sim.cpp

The FunctionalSim component is a key innovation that enables accurate timing simulation:

  • Pre-executes the ELF binary to determine actual branch outcomes
  • Provides "perfect branch prediction" for the main timing simulator
  • Executes instructions functionally (no timing model) to record taken/not-taken outcomes for each branch instruction
  • Branch outcomes are fed to the main timing simulator, eliminating misprediction penalties from the simulation
Why functional pre-simulation?

By running the binary functionally first, the simulator knows the exact outcome of every branch before the timing simulation begins. This isolates the memory hierarchy and pipeline effects from branch prediction quality, making it easier to study microarchitectural behavior independently.
