Statistics

Two main components — StatsCollector for raw event tracking and PerformanceMetrics for derived analysis — provide comprehensive simulation insights.

StatsCollector

Source file: stats/stats_collector.hpp

The StatsCollector is the central statistics gathering component during simulation. It is updated every cycle by the CPU pipeline and records the following categories of events:

  • Per-instruction latency — fetch-to-commit cycle count for each instruction
  • Cache events — hits, misses, and evictions per cache level (L1, L2, L3)
  • Memory operation counts — total loads and stores executed
  • Branch prediction results — correct and incorrect predictions
  • Pipeline stall cycles — breakdown of stall cycles by type (data hazard, cache miss, structural, etc.)
  • Functional unit usage — cycle counts for each execution unit type

Key recording methods:

record_cache_access()
Records a cache access event with hit/miss result, latency, and cache level.
record_branch()
Records a branch prediction event with predicted vs. actual outcome.
record_instruction()
Records instruction completion with per-instruction timing data.

PerformanceMetrics

Source file: stats/performance_metrics.hpp

The PerformanceMetrics component computes derived metrics from the raw statistics collected by StatsCollector. The PerformanceStats struct includes:

Metric Description
total_instructions Number of instructions simulated
total_cycles Total simulation cycles
ipc Instructions per cycle (total_instructions / total_cycles)
cpi Cycles per instruction (total_cycles / total_instructions)
Cache statistics Per level: hit count, miss count, hit rate, MPKI (misses per kilo-instructions)
Branch prediction Correct count, total count, accuracy percentage
Functional unit utilization Fraction of cycles each unit was busy
Pipeline stall distribution Breakdown of stall cycles by stall type
Instruction mix Breakdown by OpcodeType category

The metrics component also maintains two time series for trend analysis:

  • IPC time series — samples IPC every 1000 cycles for IPC trend visualization
  • Wall-clock time series — tracks real-time simulation speed (instructions per wall-clock second)

Instruction Mix Categories

Statistics are broken down by OpcodeType into the following categories:

Category Opcodes
Integer ALU Add, Sub, And, Or, Xor, Shift, Move, etc.
Integer Multiply Mul, Mneg, MADD, MSUB, etc.
Load Ldr, Ldp, Ldrb, Ldrh, Ldrsb, Ldrsw, Ldur, Prfm, etc.
Store Str, Stp, Strb, Strh, Stur, etc.
Branch B, Bl, Cbz, Cbnz, B.cond, Br, Blr, Ret, etc.
FP Fadd, Fsub, Fmul, Fdiv, Fcmp, Fmov, Fcvt, etc.
SIMD SimdAdd, SimdMul, SimdDup, SimdIns, SimdZip, etc.
System Barrier, Nop, Mrs, Msr, Sys, etc.
Crypto Aes*, Sha*, etc.
SVE SveAdd, SveMul, SveCmp, SveWhile, etc.
SME Sme* operations (Streaming SVE mode)
Instruction mix and performance correlation

The instruction mix is valuable for understanding simulation bottlenecks. For example, a high percentage of load instructions combined with a low L1 hit rate indicates a memory-bound workload. A high branch ratio with low prediction accuracy suggests control-flow-bound behavior.

Trace Output

Detailed per-instruction execution traces can be enabled via the --save-trace flag. The trace records:

  • Cycle timestamps for each pipeline stage (fetch, decode, rename, dispatch, issue, execute, writeback, commit)
  • Register values read and written by each instruction
  • Memory addresses accessed by load/store instructions
  • Branch predictions and outcomes for conditional branches
--save-trace
Enable detailed per-instruction execution trace recording in the output JSON file.
--dump-range <start>:<end>
Dump pipeline trace only for instructions in the specified range (inclusive). Example: --dump-range 100:200.
Trace file size

Enabling --save-trace can produce large output files, especially for long simulations. Use --dump-range to limit trace output to the instruction window of interest, or combine with -n to limit the total instruction count.
