Memory Hierarchy

Three-level cache hierarchy with DDR memory controller featuring configurable sizes, associativity, and latencies.

Cache Organization

The simulator models a realistic three-level cache hierarchy backed by a DDR memory controller. Each level is independently configurable via command-line flags or presets.

Level Default Size Associativity Line Size Hit Latency
L1 64 KB 4-way 64 B 4 cycles
L2 512 KB 8-way 64 B 12 cycles
L3 8 MB 16-way 64 B 40 cycles
DDR — — — 150 cycles (base)

Cache Implementation

Source files: memory/cache.hpp, memory/cache.cpp

The cache is implemented as a set-associative structure with the following characteristics:

  • LRU (Least Recently Used) replacement policy — evicts the line that has not been accessed for the longest time within each set
  • Number of sets computed as: cache_size / (associativity × line_size)
  • CacheAccessInfo struct tracks access type (hit/miss), latency, and cache level for each memory operation
  • Write-back policy for stores — modified data is written back to the next level only on eviction
  • Cache-to-cache transfers modeled in L3 for multi-core coherence scenarios

DDR Memory Controller

The DDR memory controller models realistic DRAM timing parameters:

Parameter Value
Base latency 150 cycles
Row buffer hit bonus -30 cycles (120 cycles total)
Bank conflict penalty +20 cycles
Number of banks 8 banks for parallel access

The controller models realistic DRAM timing including activate, read/write, and precharge operations. Row buffer hits (accessing the same row already open in a bank) receive a latency bonus, while bank conflicts (accessing a different row in an already-open bank) incur a penalty.

Load/Store Queue (LSQ)

Source file: memory/lsq.hpp

The Load/Store Queue sits between the execution units and the cache hierarchy, managing memory ordering and forwarding:

Parameter Default
Queue size 64 entries
Load pipeline count 2 (configurable)
Store pipeline count 1 (configurable)
Outstanding memory requests 16 (configurable)

Each LSQEntry structure tracks:

  • Virtual and physical address
  • Instruction pointer (PC)
  • Data size (byte width)
  • Completion status
  • Dependencies on preceding stores (for load-store forwarding)

The LSQ features separate load and store pipelines, allowing loads and stores to be issued independently. Store-to-load forwarding is supported when a younger load depends on an older store in the queue.

Memory Subsystem Integration

Source file: memory/memory_subsystem.hpp

The MemorySubsystem class combines all memory components into a unified interface:

  • Combines L1, L2, L3 caches and DDR controller into a single access path
  • Handles cache coherence when CHI protocol is enabled
  • record_cache_access() is called by the CPU for each memory operation
  • Returns a CacheAccessInfo struct with full timing information for the access

Cache Access Flow

Memory accesses traverse the hierarchy from L1 down to DDR, accumulating latency on each miss:

CPU Load/Store
  → L1 Cache (4 cycles if hit)
  → L2 Cache (12 cycles if hit, +4 L1 miss)
  → L3 Cache (40 cycles if hit, +12 L2 miss)
  → DDR (150 cycles base, +40 L3 miss)

On a full miss to all cache levels, the total access latency is 4 + 12 + 40 + 150 = 206 cycles. This penalty motivates the out-of-order engine to hide memory latency by executing independent instructions while memory operations are outstanding.

Non-blocking caches

Cache misses do not stall the pipeline. The LSQ tracks outstanding memory requests and wakes dependent instructions when data returns. This is critical for achieving good IPC on memory-intensive workloads.

Configuration Parameters

--l1-size <KB>
L1 data cache size in kilobytes. Default: 64. Common values: 32, 64, 128.
--l2-size <KB>
L2 cache size in kilobytes. Default: 512. Common values: 256, 512, 1024.
--l3-size <KB>
L3 (last-level) cache size in kilobytes. Default: 8192 (8 MB). Common values: 4096, 8192, 16384, 32768.
