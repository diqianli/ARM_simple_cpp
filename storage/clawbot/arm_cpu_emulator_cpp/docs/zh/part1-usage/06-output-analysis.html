输出与结果分析

了解 JSON 输出格式、性能指标以及如何解读仿真结果。

JSON 输出文件

仿真器将结果写入 JSON 文件（默认为 output.json，可通过 -o 配置）。输出包含三个主要部分：

1. 元数据 (Metadata)

仿真所使用的 CPU 配置快照，包括：

  • window_size（ROB 大小）
  • fetch_width、issue_width、commit_width
  • Cache 大小和关联度（L1、L2、L3）
  • Cache 延迟
  • CPU 频率
  • Load/Store 队列大小和流水线数量

2. 操作记录 (Operations)

每条指令的流水线时序数据。每条记录了指令经过各个 Pipeline 阶段的周期：

字段 描述
fetch_cycle指令被 Fetch 的周期
decode_cycle指令被 Decode 的周期
rename_cycle寄存器重命名完成的周期
dispatch_cycle指令被 Dispatch 到 Issue 队列的周期
issue_cycle指令被 Issue 到功能单元的周期
execute_cycle开始 Execute 的周期
memory_cycleMemory 访问完成的周期（仅 Load/Store 指令）
writeback_cycle结果 Writeback 的周期
commit_cycle指令 Retire（Commit）的周期

3. 通道数据 (Lanes)

每个周期的 Pipeline 阶段详细信息，供 Konata 可视化渲染时间线图表使用。

性能指标

使用 -j 标志可将性能指标输出到 stdout，或包含在输出 JSON 中。关键指标包括：

IPC（每周期指令数，Instructions Per Cycle）

total_instructions / total_cycles。值越高越好。对于非内存受限的工作负载，优化良好的乱序核心通常能达到 IPC > 1.0。

CPI（每指令周期数，Cycles Per Instruction）

IPC 的倒数。值越低越好。适用于与顺序处理器进行对比。

Cache 命中率

每个 Cache 层级（L1、L2、L3）的 hits / (hits + misses)。高 L1 命中率（> 90%）对性能至关重要。

MPKI（每千条指令缺失数，Misses Per Kilo Instructions）

cache_misses * 1000 / total_instructions。用于在不同仿真长度之间比较 Cache 性能的归一化指标。

分支预测准确率

correct_predictions / total_branches。现代分支预测器在典型工作负载上能达到 > 95% 的准确率。

功能单元利用率

仿真期间各功能单元类型的繁忙程度分布：

  • IntAlu — 整数算术和逻辑
  • IntMul — 整数乘法和除法
  • Load — Load 单元
  • Store — Store 单元
  • Branch — 分支解析
  • FpSimd — 浮点和 SIMD
  • System — 系统指令（屏障等）
  • Crypto — 加密指令

Pipeline 停顿分布

按类型分类的 Pipeline 停顿分布：

  • 数据冒险 (Data hazard) — 等待寄存器依赖
  • 控制冒险 (Control hazard) — 等待分支解析
  • 结构冒险 (Structural hazard) — 无可用功能单元
  • 内存 (Memory) — 等待 Cache 或内存访问

时间序列

每 1000 个周期采样一次 IPC 等指标，用于识别仿真过程中的性能阶段变化。

JSON 指标输出示例

{
  "total_instructions": 1000,

详细 Trace 选项

保存执行 Trace

使用 --save-trace 保存详细的执行 Trace，包括每个周期所有 Pipeline 阶段的状态：

./build/arm_cpu_sim -f elf program.elf --save-trace

Dump 特定指令范围

使用 --dump-range Dump 特定指令范围的 Pipeline Trace：

# Dump 指令 100 到 200 的 Pipeline Trace
./build/arm_cpu_sim -f elf program.elf --dump-range 100:200

结果分析技巧

低 IPC + 高 L1 缺失率

工作负载受内存限制。考虑增大 L1 Cache 大小、改善预取，或优化测试程序中的数据访问模式。

低 IPC + 低分支准确率

工作负载分支密集且难以预测。考虑改进分支预测器或减少测试程序中的分支密度。

功能单元利用率严重失衡

工作负载存在特定资源瓶颈。例如，如果 Load 单元已饱和而 IntAlu 空闲，则程序受限于内存带宽而非计算吞吐量。
