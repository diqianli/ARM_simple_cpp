CHI 一致性协议

实现 AMBA CHI（Coherent Hub Interface）Cache 一致性协议，用于多核 Cache 一致性建模。

仿真器实现了 AMBA CHI（Coherent Hub Interface）Cache 一致性协议，提供了 ARM 多核系统中 Cache 一致性流量的真实模型。实现位于 chi/chi_manager.hpp。

默认禁用

CHI 一致性在单核仿真中默认禁用。通过命令行参数 --enable-chi 启用，或以编程方式设置 CPUConfig::enable_chi = true。

CHI 节点类型

CHI 协议建模了三种节点类型，每种节点代表一致性层次结构中的不同角色：

1. RN-F（请求节点 - 完全一致）

代表 CPU 核心的 Cache 控制器。它向互连发出读写请求，处理以下事务：

  • 角色：向互连发出读/写请求
  • 节点 ID：通过 chi_rnf_node_id 配置（默认：0）
  • 请求类型：ReadShared、ReadClean、ReadNotSharedDirty、WriteLineUnique、MakeUnique

2. HN-F（Home 节点 - 完全一致）

充当基于目录的一致性管理器，维护 Cache 行状态信息：

  • 角色：基于目录的一致性管理器
  • 维护：Cache 行状态目录
  • 处理：Snoop 请求和数据响应
  • 节点 ID：通过 chi_hnf_node_id 配置（默认：1）
  • 目录大小：通过 chi_directory_size 配置（默认：4096 条目）

3. SN-F（Snoop 节点 - 完全一致）

处理来自 HN-F 的 Snoop 请求，为 Cache 间数据传输提供数据：

  • 角色：处理来自 HN-F 的 Snoop 请求
  • 提供：Cache 间数据传输的数据
  • 节点 ID：通过 chi_snf_node_id 配置（默认：2）

协议状态机

Cache 行在请求处理过程中在 CHI 一致性状态之间转换：

状态 描述
Invalid Cache 行不存在
Shared (S) 干净副本；可能同时存在于多个 Cache 中
Unique (U) 系统中唯一的副本；干净
UniqueDirty (UD) 系统中唯一的副本；已修改（脏）

CHI 配置参数

参数 默认值 描述
enable_chi false 启用 CHI 协议
chi_request_latency 2 周期 请求传输延迟
chi_response_latency 2 周期 响应传输延迟
chi_data_latency 4 周期 数据传输延迟
chi_snoop_latency 2 周期 Snoop 往返延迟
chi_max_pcrd_credits 16 最大 PCR（Protocol Credit）信用值
chi_max_outstanding_dbid 32 最大未完成数据缓冲区 ID 数
chi_max_retry_queue_size 64 最大重试队列条目数
chi_directory_size 4096 目录条目数

CHI 请求类型

协议支持以下请求类型，每种请求代表特定的一致性事务：

enum class ChiRequestType {
    ReadShared,         // 共享读
    ReadClean,          // 期望获取干净数据的读
    ReadNotSharedDirty, // 带修改意图的读
    WriteLineUnique,    // 写完整 Cache 行
    MakeUnique,         // 升级为 Unique
    CleanShared,        // 驱逐共享行
    MakeInvalid,        // 驱逐 Unique/脏行
};

QoS（服务质量）

CHI 实现包含流量控制机制来管理互连带宽：

  • 基于 PCR 信用的流量控制：使用 Protocol Credit（PCR）信用值限制在途请求数量，防止互连拥塞。
  • 未完成请求限制：限制并发数据缓冲区 ID 的最大数量（chi_max_outstanding_dbid），以约束资源使用。
  • 用于反压的重试队列：当信用耗尽或互连拥塞时，请求被放入重试队列（chi_max_retry_queue_size），在带宽可用时重新发出。
