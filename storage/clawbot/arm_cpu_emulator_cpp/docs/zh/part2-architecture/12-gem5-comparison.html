与 GEM5 对比

与 GEM5 仿真器的交叉验证，包括配置对齐、比较脚本和已知的架构差异。

仿真器可以配置为匹配 GEM5 的 O3_ARM_v7a CPU 模型，实现两个仿真器之间时序结果的交叉验证。这对于验证正确性和理解我们模型中的简化如何影响精度非常有用。

配置对齐

使用 --preset gem5_o3_arm_v7a 将我们的仿真器参数与 GEM5 默认值对齐：

参数 本仿真器 GEM5 O3_ARM_v7a
ROB 大小 40 40
Fetch 宽度 3 3
Issue 宽度 8 8
Commit 宽度 8 8
Load Queue 16 16
Store Queue 16 16
ICache — 32 KB, 2 路
DCache 32 KB, 2 路 32 KB, 2 路
L2 Cache 1 MB, 16 路 1 MB, 16 路
L1 命中延迟 2 周期 约 2 周期
L2 命中延迟 20 周期 约 20 周期
关键结构差异

GEM5 O3_ARM_v7a 具有分离的 ICache 和 DCache；我们的仿真器使用统一 L1。GEM5 在此配置中没有 L3；我们的仿真器保留默认的 L3。这些差异会影响 Cache 竞争建模。

比较脚本

项目包含一个 Python 脚本 scripts/compare_gem5.py，用于并排比较两个仿真器的输出：

python3 scripts/compare_gem5.py \
    --ours output.json \
    --gem5 gem5_stats.txt

比较维度

比较脚本评估以下指标：

  • 周期数：完成仿真的总模拟周期数
  • 指令数：执行的总指令数
  • IPC（每周期指令数）：整体吞吐量指标
  • 执行时间：每个仿真器的实际运行时间

已知差异

1. 分支预测

我们的仿真器使用 FunctionalSim（通过预执行实现完美预测），而 GEM5 使用可配置的分支预测器（默认：tournament）。这可能导致显著的 IPC 差异，因为完美预测消除了所有分支预测错误的代价。

2. Cache 模型

我们的 L1 Cache 是统一的（指令 + 数据），而 GEM5 具有分离的 ICache 和 DCache。这会影响 Cache 竞争建模，因为指令获取和数据访问在我们的模型中竞争相同的 Cache 空间，而在 GEM5 中是隔离的。

3. L3 Cache

我们的仿真器始终在层次结构中包含 L3 Cache，而 GEM5 O3_ARM_v7a 不配置 L3。这可能导致不同的未命中率和内存访问模式。

4. 时序粒度

我们的仿真器使用整数周期计数，而 GEM5 可能对某些操作具有亚周期时序。即使整体行为相似，这也可能导致周期计数上的微小差异。

5. 内存模型

我们的 DDR 模型包含行缓冲区时序和 Bank 冲突，而 GEM5 使用不同的内存时序模型（通常为 SimpleMemory 或 DRAMCtrl）。详细的时序参数可能有所不同。

6. 指令覆盖范围

我们基于 Capstone 的解码器覆盖约 500 个 ARM64 助记符，而 GEM5 的 ISA 解析器更为全面。Capstone 无法解码的罕见或特权指令可能被不同地处理。

带有 GEM5 变体的基准测试程序

benchmarks/ 目录包含每个基准测试的 _gem5_se 变体，预配置为 GEM5 的 SE（System-Emulation）模式。这些变体允许直接比较：

benchmarks/dhrystone/dhrystone_gem5_se
benchmarks/fibonacci/fibonacci_gem5_se
benchmarks/matmul/matmul_gem5_se
# ...以及其他基准测试

有效比较的建议

  1. 使用相同的二进制文件：我们的 _aarch64 和 _gem5_se 变体从相同源代码编译，确保指令级别的等价性。
  2. 设置相同的限制：在两个仿真器中配置相同的最大指令数（-n）和最大周期数（-c）限制。
  3. 考虑分支预测差异：我们的完美分支预测（通过 FunctionalSim）通常会比 GEM5 的 tournament 预测器显示更高的 IPC。应关注 Cache 行为以获得更具可比性的指标。
  4. 比较 Cache 指标：Cache 命中率和 MPKI（每千条指令的未命中次数）在两个仿真器之间比原始 IPC 更具可比性，因为它们受分支预测差异的影响较小。
  5. 使用比较脚本：运行 scripts/compare_gem5.py 以获得标准化的输出格式和指标对齐。
