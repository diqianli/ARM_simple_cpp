CPU 流水线

驱动仿真器中指令执行的 9 级乱序流水线。

仿真器建模了一个 9 级乱序流水线，从 Fetch 到 Commit 处理指令，中间阶段支持乱序执行。

流水线阶段

  1. Fetch (IF)：从输入源（trace 或 ELF）读取指令。每个周期最多取 fetch_width 条指令。
  2. Decode (DE)：将原始指令字节解码为内部 Instruction 结构。两条解码路径：
    • Capstone 路径（ELF 输入）：通过 Capstone v5 支持约 500 条助记符
    • 文本 trace 路径：通过自定义解析器支持约 40 条助记符
  3. Rename (RN)：寄存器重命名以消除假依赖。将架构寄存器（X0–X30、V0–V31）映射到物理寄存器。
  4. Dispatch (DI)：将重命名后的指令分派到指令窗口（ROB）。跟踪指令间的依赖关系。
  5. Issue (IS)：唤醒并从窗口中选择就绪指令。使用计分板进行依赖跟踪。每个周期最多发射 issue_width 条指令。
  6. Execute (EX)：指令在功能单元上执行。不同单元具有不同的延迟：
    • 整数 ALU：1 个周期（ADD、SUB、AND、OR 等）
    • 整数乘法：3 个周期
    • Load：4 个周期（L1 命中）
    • Store：1 个周期（地址计算）
    • 分支：1 个周期
    • FP/SIMD：3–5 个周期
    • 加密：延迟可变
  7. Memory Complete (ME)：Load 数据从 Cache 层次结构返回。Cache 未命中时可能停顿。
  8. Writeback (WB)：结果写回并广播到窗口中的依赖指令。
  9. Commit (RR)：指令按程序顺序提交。每个周期最多 commit_width 条。更新架构状态。

主循环实现

流水线由 src/cpu.cpp 中的 CPUEmulator::run() 驱动。每个周期：

  1. 提交已完成的指令（按序）
  2. 写回结果
  3. 执行就绪指令
  4. 从窗口发射新指令
  5. Dispatch/Decode/Fetch 新指令
  6. 更新统计信息

流水线示意图

Fetch → Decode → Rename → Dispatch → Issue → Execute → Memory → Writeback → Commit

流水线跟踪

PipelineTracker 记录每条指令进入/离开各阶段的时间。数据导出为 Konata JSON 格式以供可视化。每条指令的计时信息包括：

字段 说明
fetch_cycle指令进入 Fetch 阶段的周期
decode_cycle指令进入 Decode 阶段的周期
rename_cycle指令进入 Rename 阶段的周期
dispatch_cycle指令进入 Dispatch 阶段的周期
issue_cycle指令被发射执行的周期
execute_cycle指令开始执行的周期
memory_cycle内存操作完成的周期
writeback_cycle结果写回的周期
commit_cycle指令提交的周期
